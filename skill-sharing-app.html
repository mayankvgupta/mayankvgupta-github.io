01 — Context

Learning happens outside the classroom. There was nowhere to record it.

Pratt Beta Space commissioned an MVP for a skill sharing platform — a mobile application that would let students log hours spent learning skills outside formal coursework, authenticate their progress, and connect with peers working on the same things.

The goal wasn't to build everything. It was to build enough to start testing. The brief called for an MVP scope: four core flows, a coherent visual system, and a prototype ready for user testing with the development team. I worked alongside a product developer and product manager on a two-person design team, guided by Prof. Basem Aly, over eight weeks.

02 — Process

User flow first. Then screens.

Before any sketch reached the screen, the team mapped user flows — starting from primary student goals and iterating through multiple flow versions with a student persona in mind. This grounded the screen prioritisation: login, registration, skill adding, and hours logging were identified as the four non-negotiable MVP flows.

Initial Sketches

Paper Prototyping

Initial sketches · Primary flows · Mapping screens before committing to digital

Paper prototype testing with 5 participants surfaced two issues that shaped the wireframe phase: the registration flow's password requirements needed clearer progressive disclosure — three iterations were explored before landing on an approach that worked — and the onboarding sequence needed reordering based on how students actually thought about joining a skill platform.

03 — Design

From wireframes to high-fidelity. Four flows, one system.

Low-fidelity wireframes were built after the paper prototype feedback was resolved. Working in parallel with a shared style guide ensured visual consistency across screens owned by different team members — a deliberate process choice that kept the prototype cohesive despite the parallel workstream.

Final wireframes · Core screens · Login, registration, skill adding, hours logging

Login flow
Minimal entry point with clear registration pathway for new users. Onboarding designed to feel approachable rather than gate-like.
Registration flow
Three iterations on password requirement UX before settling on inline progressive validation — the most-tested screen in the prototype.
Skill + Hours flows
The core product loop: add a skill, log hours against it, see progress accumulate. Designed for students logging learning time across multiple disciplines.

High-fidelity prototype · All four MVP flows · Adobe XD

04 — Outcome

MVP delivered. Development handoff completed.

The high-fidelity prototype covered all four core flows — login, registration, skill adding, and hours logging — and was handed off to the development team as the design foundation for the initial build. The prototype was ready for the first round of user testing with actual students as soon as development produced a functional version.

Working within an MVP constraint was a useful discipline: it forced prioritisation decisions early and kept the design focused on the flows that mattered most for initial validation. Features that would have been valuable — peer browsing, skill matching, progress authentication — were scoped for later iterations once the core loop was tested.

The project was also my first experience working directly alongside a product developer and product manager — navigating handoff requirements, understanding technical constraints, and keeping the design system tight enough for a developer to implement without ambiguity. That context shaped how I document and present design work.
